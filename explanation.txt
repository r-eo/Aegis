────────────────────────────────────────────────────────────────────────────────
1. PROJECT OVERVIEW
────────────────────────────────────────────────────────────────────────────────

Project Aegis is a real-time predictive maintenance platform for industrial
machinery. It uses a smartphone's built-in accelerometer sensor to stream
vibration data to a cloud backend, which runs an unsupervised machine learning
algorithm (Isolation Forest) to detect anomalies and predict potential machine
failures before they happen.

The system has two deployable components:

  1. BACKEND (Core Engine)  → Hosted on Render (Python / FastAPI)
  2. FRONTEND (Command Center) → Hosted on Vercel (React / Vite)

A smartphone opens the Vercel URL, grants accelerometer permission, and starts
streaming vibration data at 20 Hz over a WebSocket to the Render backend. The
backend processes the data, runs the ML model, and broadcasts anomaly scores
back to all connected dashboard clients in real time.

3. DATASETS
────────────────────────────────────────────────────────────────────────────────

Two real-world bearing failure datasets are used to pre-train the model.

─── 3.1 CWRU Dataset (Case Western Reserve University) ─────────────────────────

File    : datasets/cwru_unsupervised_features.csv
Rows    : 2,301
Source  : CWRU Bearing Data Center (most widely used bearing fault dataset)

Description:
  The CWRU dataset contains accelerometer readings from drive-end and fan-end
  bearings under different fault conditions (normal, ball fault, inner race
  fault, outer race fault) at various motor loads (0–3 HP).

Columns (pre-extracted statistical features per 2-second window):
  - max       : Maximum acceleration value in the window
  - min       : Minimum acceleration value in the window
  - mean      : Mean (DC offset) of the signal
  - sd        : Standard deviation of the signal
  - rms       : Root Mean Square of the signal
  - skewness  : Statistical skewness (asymmetry measure)
  - kurtosis  : Statistical kurtosis (tail/impulsiveness measure)
  - crest     : Crest factor (peak / RMS ratio — key fault indicator)
  - form      : Form factor (RMS / mean_abs — shape descriptor)

Features used for ML training from CWRU:
  - max  → renamed to max_vibration
  - rms  → renamed to rms_vibration

The CWRU dataset contains a natural mix of healthy and faulty bearing data,
making it ideal for training an unsupervised anomaly detector.

─── 3.2 NASA IMS Bearing Dataset ───────────────────────────────────────────────

File    : datasets/nasa_test2_features.csv
Rows    : 985
Source  : NASA IMS (Intelligent Maintenance Systems) Bearing Dataset, Test 2

Description:
  The NASA dataset is a run-to-failure experiment. Bearings were run continuously
  until failure. The test2 subset shows a clear degradation trajectory — starting
  from healthy operation and progressing to failure over several days.

Columns (pre-extracted):
  - timestamp_id    : Date/time string (e.g. 2004.02.12.10.32.39)
  - max_vibration   : Maximum acceleration in window (g)
  - rms_vibration   : RMS acceleration in window (g)

This dataset is especially valuable because it shows the temporal degradation
pattern — RMS and MAX values slowly rise as the bearing approaches failure.
The final ~280 rows show catastrophic vibration increase (failure region).

─── 3.3 Combined Training Set ───────────────────────────────────────────────────

Total samples : 3,286 (2,301 CWRU + 985 NASA)
Features used : max_vibration, rms_vibration
Scaling       : StandardScaler (zero mean, unit variance) applied before training


────────────────────────────────────────────────────────────────────────────────
4. MACHINE LEARNING MODEL — ISOLATION FOREST
────────────────────────────────────────────────────────────────────────────────

─── 4.1 Why Isolation Forest? ───────────────────────────────────────────────────

Isolation Forest is an UNSUPERVISED anomaly detection algorithm. This is the
correct choice for predictive maintenance because:

  1. NO LABELS NEEDED: We don't need fault/no-fault labels. The model learns
     what "normal" vibration looks like and flags anything different.

  2. WORKS ON RAW FEATURES: It operates on simple numerical features (max, rms)
     rather than raw time series.

  3. EFFICIENT: O(n log n) training time, O(log n) inference. Fast enough for
     real-time use at 20 Hz.

  4. INTERPRETABLE: The anomaly score has clear meaning (0 = normal, 1 = fault).

  5. ROBUST: Handles high-dimensional data without distributional assumptions.

─── 4.2 Algorithm Explanation ───────────────────────────────────────────────────

Isolation Forest works by randomly partitioning data:

  1. Build many decision trees (n_estimators=200) where each tree randomly
     selects a feature and a random split value between min and max.

  2. Normal points require MANY splits to isolate (they cluster together).
     Anomalies require FEW splits to isolate (they are sparse/extreme).

  3. The anomaly score is based on the average path length to isolate a point
     across all trees. Shorter path → more anomalous.

  4. score_samples() returns a raw score near 0 for normal, negative for anomalies.

─── 4.3 Model Hyperparameters ───────────────────────────────────────────────────

  n_estimators    = 200       (number of isolation trees — more = stabler estimates)
  contamination   = 0.05      (expected fraction of anomalies = 5%)
  max_samples     = "auto"    (uses min(256, n_samples) samples per tree)
  random_state    = 42        (for reproducibility)
  n_jobs          = -1        (use all CPU cores for training)

The contamination=0.05 means the model expects ~5% of training samples to be
anomalous, which aligns with the known fault ratio in the CWRU dataset.

─── 4.4 Score Normalization ─────────────────────────────────────────────────────

Raw output from IsolationForest.score_samples() is in the range [-1.0, ~0.0].

We map this to a clean [0.0, 1.0] anomaly score using:

  1. Clip raw score to [-1.0, 0.1]
  2. Normalize: anomaly_score = (raw_score - 0.1) / (-1.0 - 0.1)
  3. Clip result to [0.0, 1.0]

This gives:
  - raw ≈ 0.0   → anomaly_score ≈ 0.0  (perfectly normal)
  - raw ≈ -0.5  → anomaly_score ≈ 0.5  (moderate anomaly)
  - raw ≈ -1.0  → anomaly_score ≈ 1.0  (severe anomaly)

─── 4.5 Alert Thresholds ────────────────────────────────────────────────────────

  anomaly_score > 0.7  → CRITICAL  (immediate alert, red UI)
  anomaly_score > 0.4  → WARNING   (caution, yellow UI)
  anomaly_score ≤ 0.4  → HEALTHY   (normal operation, green UI)

These thresholds are based on the distribution of the training data and can be
tuned in core_engine/main.py.

─── 4.6 Model Persistence ────────────────────────────────────────────────────────

The trained model is saved to disk on first startup using joblib:

  core_engine/trained_model.joblib   ← IsolationForest model
  core_engine/scaler.joblib          ← StandardScaler (fit on training data)
  core_engine/pca_model.joblib       ← PCA model (2 components)

On subsequent startups, the saved models are loaded (fast). The .joblib files
are gitignored so they are regenerated fresh on every Render deployment.

5. PRINCIPAL COMPONENT ANALYSIS (PCA)
────────────────────────────────────────────────────────────────────────────────

─── 5.1 Purpose ─────────────────────────────────────────────────────────────────

PCA is used here for VISUALIZATION, not for anomaly detection. Since we only
have 2 features (max_vibration, rms_vibration), PCA mostly shows the variance
structure of the data, but if more features were added (e.g. all 9 CWRU columns),
PCA would reduce them to 2D for the scatter plot.

─── 5.2 Configuration ───────────────────────────────────────────────────────────

  n_components = 2
  Applied AFTER StandardScaler (so features are on equal scale)

─── 5.3 Dashboard Visualization ─────────────────────────────────────────────────

The PCA scatter plot in the dashboard shows:
  - Green dots  : CWRU normal samples
  - Red dots    : CWRU anomaly samples (flagged by IsolationForest)
  - Purple dots : NASA normal samples
  - Orange dots : NASA anomaly samples
  - White star  : LIVE point (current incoming telemetry projected into PCA space)

─── 5.4 REST Endpoint ────────────────────────────────────────────────────────────

GET /api/pca-analytics

Returns:
  {
    "points": [{"pc1": ..., "pc2": ..., "anomaly_score": ..., "is_anomaly": ..., "source": "cwru|nasa"}, ...],
    "explained_variance": [0.xx, 0.xx],
    "total_samples": 3286,
    "anomaly_count": 164
  }

Max 500 points are returned (random sample) for dashboard performance.


────────────────────────────────────────────────────────────────────────────────
6. TELEMETRY PROCESSING PIPELINE
────────────────────────────────────────────────────────────────────────────────

─── 6.1 Signal Flow ─────────────────────────────────────────────────────────────

Phone accelerometer
  ↓ raw X, Y, Z acceleration values (m/s² or g, depending on device)
Vector Magnitude = sqrt(X² + Y² + Z²)
  ↓ scalar magnitude per sample
Rolling Window Buffer (deque, maxlen=40 samples = 2 seconds at 20 Hz)
  ↓ last 40 magnitudes
Feature Extraction:
  max_vibration = max(buffer)
  rms_vibration = sqrt(mean(buffer²))
  ↓ 2 features
StandardScaler.transform([[max_vib, rms_vib]])
  ↓ scaled features
IsolationForest.score_samples()
  ↓ raw anomaly score
Normalization → [0.0, 1.0] anomaly score
  ↓
Alert classification (HEALTHY / WARNING / CRITICAL)
  ↓
Broadcast to all connected WebSocket clients

─── 6.2 TelemetryProcessor Class ────────────────────────────────────────────────

File: core_engine/telemetry_processor.py

  - Uses collections.deque(maxlen=40) as a circular buffer
  - Window size: 40 samples
  - At 20 Hz sampling rate: 40 / 20 = 2 second window
  - Vector magnitude computed per incoming sample
  - Feature extraction only possible when buffer has ≥ 1 sample

─── 6.3 Sampling Rate ────────────────────────────────────────────────────────────

The frontend sends data every 50ms (SEND_INTERVAL_MS = 50), which gives 20 Hz.
This is the standard rate for vibration analysis in industrial applications.
The buffer fills in 2 seconds (40 samples × 50ms).


────────────────────────────────────────────────────────────────────────────────
7. BACKEND API (FastAPI)
────────────────────────────────────────────────────────────────────────────────

─── 7.1 Startup (Lifespan) ──────────────────────────────────────────────────────

On server startup (using FastAPI's lifespan context manager):
  1. TelemetryProcessor is initialized (window_size=40)
  2. MLPipeline is initialized:
     a. Tries to load saved .joblib models from disk
     b. If not found, loads both CSVs from datasets/ folder
     c. Scales data with StandardScaler
     d. Trains IsolationForest (200 trees, contamination=0.05)
     e. Fits PCA (2 components)
     f. Saves models to .joblib files
  3. Server is ready to accept connections

─── 7.2 WebSocket Endpoint ──────────────────────────────────────────────────────

Path    : /ws/telemetry
Protocol: WebSocket (WSS in production)

Incoming message format (JSON string):
  {
    "vib_x": 0.123,
    "vib_y": -0.456,
    "vib_z": 9.81,
    "timestamp": 1713198210.5
  }

Outgoing broadcast format (JSON, sent to ALL connected clients):
  {
    "timestamp": 1713198210.5,
    "anomaly_score": 0.142,
    "max_vibration": 0.523,
    "rms_vibration": 0.124,
    "alert_status": "HEALTHY",
    "pca": {
      "pc1": -1.23,
      "pc2": 0.45
    }
  }

During buffer fill phase (first 2 seconds):
  { "status": "buffering", "buffer_size": 12, "timestamp": ... }

─── 7.3 REST Endpoints ──────────────────────────────────────────────────────────

GET  /                    → Health check (service name, version)
GET  /health              → Detailed health (model_ready, ws_connections, timestamp)
GET  /api/model-info      → Model metadata (type, params, training stats)
GET  /api/pca-analytics   → PCA scatter plot data (max 500 sampled points)
GET  /api/dataset-stats   → Training data statistics (min/max/mean/std)
POST /api/score           → Score a single reading (REST, no WebSocket needed)

─── 7.4 Connection Manager ──────────────────────────────────────────────────────

The ConnectionManager class tracks all active WebSocket connections in a list.
When a new anomaly score is computed, it is BROADCAST to ALL connections.

This means:
  - The phone sends data → backend processes → result goes to phone + dashboard
  - Multiple dashboard viewers can connect simultaneously
  - Dead connections are automatically cleaned up on send failure

─── 7.5 CORS Configuration ──────────────────────────────────────────────────────

  allow_origins=["*"]     → All origins allowed (fine for MVP, tighten in prod)
  allow_credentials=True
  allow_methods=["*"]
  allow_headers=["*"]

─── 8.6 useAccelerometer Hook ───────────────────────────────────────────────────

File: src/hooks/useAccelerometer.js

  States:
    permission   : 'unknown' | 'granted' | 'denied' | 'unavailable'
    isStreaming  : boolean

  How it works:
    1. requestPermission() is called on button click (MUST be user gesture on iOS)
    2. If DeviceMotionEvent.requestPermission is a function (iOS 13+):
       → calls it, awaits result ('granted' or 'denied')
    3. If not (Android / Desktop):
       → auto-grants and proceeds
    4. On grant: opens a WebSocket, attaches devicemotion event listener
    5. setInterval (50ms = 20 Hz) sends { vib_x, vib_y, vib_z, timestamp }
    6. Uses accelerationIncludingGravity for raw sensor values

  Note: Works ONLY on HTTPS origins. Both Vercel and Render provide HTTPS.
